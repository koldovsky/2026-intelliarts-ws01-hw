# A/B Experiment

Підзадача для порівняння: реалізація `packages/element/src/textCodeBlock.ts` (parser + helper для fenced code blocks).

| Варіант (інструмент / модель) | Час | Токени | Якість | Переробки | Моменти |
|---|---|---|---|---|---|
| Run A: opencode / big-pickle | ~3 хв | н/д | Чистий regex, покрито edge cases, одразу з тестами | 0 | Добре впорався з маленьким isolated task; одразу додав tests |
| Run B: opencode / openai/gpt-5.4 | ~2 хв | н/д | Дуже мінімальна реалізація, без зайвого коду, легко верифікується | 0 | Найкраще спрацював на вузькому MVP-scoped task, де важлива стислість |

## Висновок

- **Run A** Не справився із задачею без коректного proposal. Після цього спрацював непогано.
- **Run B** краще підходить для дуже вузьких підзадач, де головне — мінімальна правильна реалізація без розростання рішення.
- Для цього конкретного завдання різниця в якості невелика. Але сам підхід був створений за допомогою openai/gpt-5.4.
- Найбільше обмеження експерименту: підзадача була дуже маленька, тому різниця між моделями проявляється слабше, ніж на більшому capability slice.
